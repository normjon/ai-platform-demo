Design Principle: Inversion of Control
This architecture is a deliberate implementation of the Inversion of Control (IoC) principle applied to AI agent design.
The inversion:
* Traditional agent: the agent controls its own domain behavior — when to retrieve, how to respond, when to escalate. Domain logic lives in agent code.
* This architecture: the skill controls domain behavior. The agent is a generic executor that receives behavior through injection. Domain logic lives in SKILL.md files.
The IoC mapping:
IoC Concept	This Architecture
Component	Generic agent shell (Strands Agent)
External definition	Skill (SKILL.md)
IoC container	skills_loader.py (reads, composes, injects)
Dynamic IoC	Orchestrator (runtime skill assignment)
Strategy Pattern	Same agent + different skill = different behavior
Active constraints — every section must reflect IoC:
* Skills sections: explicitly name what control was inverted from the agent to the skill. What did agent.py decide before that SKILL.md decides now?
* Agent sections: explicitly name what was removed from the agent and why. The agent shell must be described as containing zero domain knowledge.
* Orchestrator sections: explicitly describe the dynamic injection decisions the orchestrator makes at runtime — which skill goes to which agent, and why.
* Implementation sections: describe how IoC is enforced structurally — version pinning, one domain skill per agent, no domain logic in agent code.
The architectural claim this document must support:
"Domain behavior is fully controlled by skills. The agent shell contains zero domain knowledge. Adding a new domain capability requires no agent code changes — only a new skill."
Section 7 validates this claim explicitly. Every prior section must build toward making it true.

The Architectural Model
Three layers. Each layer has a single, clear responsibility.
Layer 1 — Skills (Intelligence)
Skills provide the specialized knowledge and procedural intelligence required to complete a specific task. A skill is a SKILL.md file that teaches an agent:
* What domain knowledge applies to this task
* When and how to use available tools
* What good output looks like
* What constraints must never be violated
Skills are authored by domain experts. Skills are the unit of domain intelligence. Skills do not execute — they instruct.
Layer 2 — Agents (Execution)
Agents execute tasks. An agent is a generic Strands execution tier that:
* Loads one domain skill injected at startup
* Runs the Strands agentic loop (reason → tool call → respond)
* Has access only to the tools its skill declares
Agents have no domain knowledge of their own. All domain intelligence comes from the injected skill. The same generic agent shell becomes an HR agent, a Finance agent, or a Legal agent depending solely on which skill is loaded. One domain skill per agent — enforced.
Layer 3 — Orchestrator (Routing)
The orchestrator is the universal entry point for every request — simple or complex. A single-skill request still flows through the orchestrator. The orchestrator is never optional.
It:
* Receives every user request
* Determines which agent tier is needed (conversational, workflow, etc.)
* Determines which skill to inject into that agent
* Dispatches to one or multiple sub-agents (serially or in parallel)
* Assembles the final response from sub-agent results
* Enforces all global rules: auth, audit logging, rate limiting, PII detection
The orchestrator itself is driven by a platform skill — a routing and coordination skill owned by the platform team. The orchestrator has no domain knowledge of its own.


User Request
      ↓
Orchestrator (platform/orchestrator skill — always)
      ↓ determines: which agent tier + which domain skill
Sub-Agent (one domain skill injected at startup)
      ↓ executes: Strands loop + declared tools
Orchestrator assembles final response
      ↓
User Response

The Current HR Assistant
The existing HR Assistant is a boto3-based implementation with three behaviors hardcoded in agent.py:
1. KB Retrieval — calls bedrock-agent-runtime.retrieve() to search the HR knowledge base before answering any policy question
2. Grounded Q&A — answers HR policy questions citing retrieved documents
3. Escalation — routes to a human HR representative when the question is outside the agent's scope or requires human judgment
These three behaviors are currently encoded in ~130 lines of manual loop logic. The goal is to extract them into skills so the agent loop becomes generic and the domain intelligence lives in versioned SKILL.md files.
The IoC framing of this migration:
* Behavior 1 (retrieval) — control moves from agent.py to hr/policy-lookup skill
* Behavior 2 (grounded Q&A) — control moves from agent.py to hr/policy-lookup skill
* Behavior 3 (escalation) — control moves from agent.py to hr/escalation-procedure skill
